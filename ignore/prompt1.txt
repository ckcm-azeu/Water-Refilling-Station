# PROMPT #1 — GENERAL FOUNDATION, SYSTEM CORE, SHARED ASSETS, ALL APIs & PUBLIC PAGES

You are building a complete web application called **"Azeu Water Station"** — a Philippine-based Water Refilling Station management system. This is a school project. You will generate ALL code files for the foundation, shared assets, configuration, all API endpoints, authentication system, public pages, and the logging/debugging system. Role-specific frontend pages (customer/, rider/, staff/, admin/ folders) will be provided in separate follow-up prompts — do NOT create those folders or their contents now.

---

## PROJECT OVERVIEW

Azeu Water Station is a web-based order and delivery management system for a water refilling station. Customers register, place orders for water and other station products, and have them delivered by riders or pick them up at the store. Staff and Admin manage accounts, confirm orders, assign riders, and monitor operations.

**4 Roles exist:**
- **Customer** — Registers, places orders, tracks delivery, receives items
- **Rider** — Receives delivery assignments, manages delivery priority, marks deliveries
- **Staff** — Approves accounts, confirms orders, assigns riders, views inventory (read-only), views rider statistics
- **Admin** — Everything Staff can do + inventory CRUD, analytics, session logs, full system settings. A **Super Admin** is a special Admin seeded into the database that cannot be deleted and is the only one who can create other Admin accounts or modify itself.

## TECH STACK & CONSTRAINTS (STRICT)

- **PHP** — No frameworks (no Laravel, no CodeIgniter, nothing). Pure PHP only.
- **MySQL** via **XAMPP** — Database name: `station_azeu`
- **HTML / CSS / JavaScript** — No CSS frameworks (no Bootstrap, no Tailwind). All CSS is hand-written.
- **Session-based authentication** — PHP `$_SESSION`. No JWT, no token-based auth.
- **If a database table does not exist, it MUST be automatically created** by `config/database.php` on connection. Never throw an error for missing tables.
- **All role code is separated into different folders** (`customer/`, `rider/`, `staff/`, `admin/`). Code is duplicated per role, not shared via includes for role pages.
- Every page-specific **CSS** goes in a separate CSS file inside that role's `css/` folder.
- Every page-specific **JS** goes in a separate JS file inside that role's `js/` folder.
- **Do NOT use `echo` to output large HTML blocks in PHP.** Use PHP for logic and embedded `<?php ?>` tags within HTML structure.

## ORDER LIFECYCLE — STATUS STATE MACHINES

### Delivery Orders (Cash on Delivery)

```
PENDING ──► CONFIRMED ──► ASSIGNED ──► ON DELIVERY ──► DELIVERED ──► ACCEPTED (optional)
   │             │             │              │
   │             │             │              └── Cancelled (by Staff/Admin with reason)
   │             │             └── Cancelled (by Rider → reverts to Confirmed; or Staff/Admin)
   │             └── Cancelled (by Staff/Admin with reason)
   └── Cancelled (by Customer with reason; or Staff/Admin)
```

**Key rules:**
- **Customer** can only cancel at `Pending` status. Cancellation increments their `cancellation_count`. If count reaches `max_cancellation` setting, account status becomes `flagged` and customer cannot place new orders until they submit an appeal and it's approved.
- **Rider** can cancel at `Assigned` or `On Delivery`. When rider cancels, order reverts to `Confirmed` (unassigned) so Staff can reassign.
- **Staff/Admin** can cancel at ANY status with a mandatory reason.
- **Stock is decremented** only when order moves from `Pending` → `Confirmed`.
- **Stock is restored** when any confirmed/assigned/on-delivery order is `Cancelled`.
- **Stock is NOT restored** when a `Pending` order is cancelled (stock was never decremented).
- **Expected Delivery Date** is automatically set to TODAY when rider marks order as `On Delivery`.
- **Delivered** → the order's `delivered_at` timestamp is set. Income is counted.
- **Accepted** is optional — customer can mark as "Accepted/Received" after delivery. Sets `customer_confirmed = 1` and `customer_confirmed_at`. Does NOT affect income counting.
- **Delivery fee** is the flat rate from the `delivery_fee` setting. Not zero.

### Pickup Orders (Pickup at Store)

```
PENDING ──► CONFIRMED ──► READY FOR PICKUP ──► PICKED UP ──► ACCEPTED (optional)
   │             │
   │             └── Cancelled (by Staff/Admin)
   └── Cancelled (by Customer or Staff/Admin)
```

**Key rules:**
- No rider assigned. `rider_id` stays NULL.
- Delivery fee = ₱0.00.
- Staff/Admin moves from `Confirmed` → `Ready for Pickup`.
- Staff/Admin marks `Picked Up` when customer collects at store. Income is counted.
- Customer can optionally mark as `Accepted` after `Picked Up`.

## NOTIFICATION SYSTEM DESIGN

Two types of notifications exist in this system:

### Type 1: SweetAlert2 Dialogs (Immediate/Blocking)
Used for: confirmation prompts ("Are you sure?"), success/error messages after actions, form dialogs (edit, cancel reason, rider selection, etc.). Triggered by user actions. These are NOT stored in the database.

### Type 2: Bell Notifications (Persistent/Stored)
Stored in `notifications` table. Bell icon in the navigation header with an unread count badge. Clicking the bell shows a dropdown listing recent 10 notifications sorted by newest. Each notification shows: icon + title + time ago + clickable to redirect to relevant page. Has a "Mark All Read" button. Refreshed on page load (no AJAX polling — user sees updates when they refresh or navigate).

## KEY ALGORITHMS

### 1. Auto-Assign Rider Algorithm

```
FUNCTION autoAssignRider(order_id):
    1. Get all riders WHERE role='rider' AND status='active' AND is_available=1 AND deleted_at IS NULL
    2. For each rider, COUNT their orders WHERE status IN ('assigned','on_delivery') → this is "current_load"
    3. Sort riders by current_load ASC (least loaded first)
    4. If tie in load, sort by rider id ASC (first registered gets priority)
    5. If NO available riders found → return NULL (order stays Confirmed, staff can add comment "Waiting for available rider")
    6. Select the top rider → UPDATE orders SET rider_id=rider, status='assigned'
    7. Create notification for the assigned rider
    8. Create delivery_priority entry for this rider+order
    RETURN assigned rider_id or NULL
```

### 2. Monthly Cancellation Reset (Runs on every page load in session.php)

```
For customers WHERE cancellation_reset_date <= TODAY:
    SET cancellation_count = 0
    SET cancellation_reset_date = first day of next month
    NOTE: Status stays 'flagged' if flagged — requires appeal to unflag
```

### 3. Pending Account Expiry Cleanup (Runs on every page load in session.php)

```
DELETE FROM users WHERE status='pending' AND DATEDIFF(NOW(), created_at) > {pending_expiry_days from settings}
Also delete related user_preferences rows.
```

### 4. Stock Validation at Checkout

```
FUNCTION validateStock(cart_items[]):
    removed_items = []
    For each item in cart_items:
        current_stock = SELECT stock_count FROM inventory WHERE id=item.inventory_id AND status='active'
        If item not found OR status != 'active' → add to removed_items
        Else if current_stock < item.quantity → add to removed_items (out of stock)
        Else if current_stock == 0 → add to removed_items
    RETURN { valid_items, removed_items }
    Show dialog to customer listing removed items if any exist.
```

### 5. Password Encryption Toggle Migration

```
WHEN admin toggles "Encrypt DB Passwords":
    If turning ON (0 → 1):
        SELECT all users
        For each user: UPDATE password = encrypt(current_password, ENCRYPTION_KEY)
    If turning OFF (1 → 0):
        SELECT all users
        For each user: UPDATE password = decrypt(current_password, ENCRYPTION_KEY)
```

### 6. Login Lockout

```
ON failed login:
    INCREMENT login_attempts for that user
    If login_attempts >= max_login_attempts setting:
        SET login_locked_until = NOW() + login_lockout_minutes minutes
        Show: "Account temporarily locked. Try again in X minutes."
ON successful login:
    RESET login_attempts = 0
    SET login_locked_until = NULL
ON login attempt while locked:
    If login_locked_until > NOW():
        Show: "Account is locked. Try again at {time}."
        Do NOT increment attempts further.
    Else:
        Lock expired → reset login_attempts = 0, login_locked_until = NULL, allow login attempt
```

### `config/AESCrypt.php`

Create a class or two standalone functions:
- `encrypt($plainText, $key)` — Returns AES-256-CBC encrypted string (base64 encoded with IV prepended).
- `decrypt($encryptedText, $key)` — Returns decrypted plain text.

Use PHP's `openssl_encrypt` / `openssl_decrypt` with `AES-256-CBC` method. Generate a random IV for each encryption, prepend the IV to the ciphertext, base64 encode the whole thing. On decrypt, extract the IV from the front, then decrypt.

## ALL API ENDPOINTS — DETAILED SPECIFICATIONS

**Every API endpoint must:**
1. Include `config/session.php` (which includes database, functions, logger)
2. Log function entry with `logger_function_entry()`
3. Validate CSRF token on POST requests
4. Check authentication (except public endpoints: login, register, forgot_password, reset_password, get_receipt)
5. Check role permissions
6. Validate and sanitize all input
7. Return JSON responses via `json_response()`
8. Log all significant actions and errors
9. Log function exit with `logger_function_exit()`

### `api/auth/login.php` (POST)

**Input:** `username`, `password`, `csrf_token`

**Logic:**
1. Log function entry
2. Check if maintenance mode ON — if yes and user role (after lookup) is not admin/super_admin → return error "System is under maintenance"
3. Look up user by username WHERE `status != 'deleted'`
4. If not found → log warning, return error "Invalid credentials"
5. Check if account is locked (`login_locked_until > NOW()`) → return error with remaining time
6. Get `encrypt_passwords` setting → if ON, decrypt stored password with AESCrypt, compare; if OFF, compare plain
7. If password wrong → increment `login_attempts`, check if it reached `max_login_attempts` → if yes, set `login_locked_until`, log warning, return error "Account locked"
8. If password correct:
   - Reset `login_attempts = 0`, `login_locked_until = NULL`
   - Create session: `$_SESSION['user_id']`, `$_SESSION['username']`, `$_SESSION['role']`, `$_SESSION['full_name']`, `$_SESSION['status']`
   - Load `user_preferences` → store `dark_mode` in session
   - Check `force_dark_mode` setting → override if ON
   - Log to `session_logs` table (action='login', IP, username snapshot, role snapshot)
   - Log info "User logged in successfully"
   - Return: `{success: true, role: "...", redirect: "/{role}/dashboard.php"}`

### `api/auth/logout.php` (POST)

**Input:** `csrf_token`

**Logic:**
1. Log to `session_logs` (action='logout')
2. Log info "User logged out"
3. `session_destroy()`
4. Return: `{success: true, redirect: "/index.php"}`

### `api/auth/register.php` (POST)

**Input:** `full_name`, `username`, `email`, `full_address`, `phone`, `password`, `confirm_password`, `csrf_token`

**Logic:**
1. Validate all fields (non-empty, email format, username unique, password >= 6 chars, passwords match)
2. Check `encrypt_passwords` setting → encrypt password if ON
3. Insert into `users`: role=customer, status=pending, cancellation_reset_date=first_of_next_month
4. Insert into `customer_addresses`: customer_id, label='Home', full_address, is_default=1
5. Insert into `user_preferences`: user_id, dark_mode=0
6. Log info "New customer registered"
7. Return success

### `api/auth/forgot_password.php` (POST)

**Input:** `email`, `csrf_token`

**Logic:**
1. Look up user WHERE email AND role='customer' AND status='active'
2. If found:
   - Generate random token (64 chars hex)
   - Insert into `password_resets`: email, token, expires_at=NOW()+1hour
   - Send email via PHPMailer with reset link: `{base_url}/reset_password.php?token={token}`
   - Log info "Password reset email sent"
3. Always return: `{success: true, message: "If this email is registered, a reset link has been sent."}`

### `api/auth/reset_password.php` (POST)

**Input:** `token`, `password`, `confirm_password`, `csrf_token`

**Logic:**
1. Validate token in `password_resets`: exists, not used, not expired
2. If invalid → return error
3. Validate passwords (>= 6 chars, match)
4. Find user by email from the reset record
5. Update user password (encrypt if setting ON)
6. Mark token as used
7. Log info "Password reset completed"
8. Return success

### `api/orders/place_order.php` (POST)

**Allowed roles:** customer (active, not flagged)

**Input:** `items` (array of {inventory_id, quantity}), `payment_type` (cod/pickup), `delivery_address` (text or address_id), `order_notes` (optional), `csrf_token`

**Logic:**
1. Validate customer status is 'active' (not flagged, not pending)
2. Validate payment_type is 'cod' or 'pickup' (reject 'online' with message "Online payment coming soon")
3. Determine delivery_type: 'delivery' if cod, 'pickup' if pickup
4. Validate items array is non-empty
5. **Real-time stock validation:** For each item, check inventory WHERE id=item.inventory_id AND status='active':
   - If item not found or inactive or out_of_stock or stock_count < requested_quantity → add to removed_items list
6. If any items removed → return `{success: false, removed_items: [...], message: "Some items are unavailable"}` (frontend handles showing which items were removed)
7. If ALL items removed → return error "All items are unavailable"
8. Determine delivery address: if delivery_type='delivery', require address. If an `address_id` is provided, look it up; if raw text, use that. If pickup, set NULL.
9. Get `delivery_fee` setting → set to 0 if pickup
10. Calculate subtotals: for each item, `item_price × quantity`. Sum all for order subtotal.
11. Calculate total: `subtotal + delivery_fee`
12. Generate `receipt_token` (random 64 char hex)
13. **Do NOT decrement stock yet** (stock is decremented only at Confirmation)
14. Insert into `orders`: all fields
15. Insert into `order_items`: snapshot each item's current name, icon, price from inventory
16. Create notification for all active staff + admin: "New order #{id} from {customer_username}"
17. Log info "Order created"
18. Return: `{success: true, order_id: X, receipt_token: "xxx"}`

### `api/orders/get_orders.php` (GET)

**Allowed roles:** all authenticated users (scoped by role)

**Input (query params):** `status` (filter), `search` (text), `page` (pagination), `date_from`, `date_to`, `role_scope` (for staff/admin: all orders; for customer: own orders; for rider: assigned orders)

**Logic:**
1. Based on role:
   - Customer: only return orders WHERE `customer_id = session.user_id`
   - Rider: only return orders WHERE `rider_id = session.user_id`
   - Staff/Admin: return all orders
2. Apply status filter if provided
3. Apply search filter (by order ID, customer name, item name)
4. Apply date range filter
5. Join with `order_items` to get item names/icons
6. Join with `users` (customer and rider) to get names
7. Paginate: LIMIT {ITEMS_PER_PAGE} OFFSET {(page-1)*ITEMS_PER_PAGE}
8. Return total count + paginated results

### `api/orders/update_status.php` (POST)

**Allowed roles:** rider (own orders: assigned→on_delivery, on_delivery→delivered), staff, admin

**Input:** `order_id`, `new_status`, `comment` (optional staff comment), `csrf_token`

**Logic:**
1. Fetch order by ID
2. Validate role permissions for this status transition:
   - Rider can only: assigned→on_delivery, on_delivery→delivered (for their own orders)
   - Staff/Admin can: any transition
3. If `new_status = on_delivery`: set `expected_delivery_date = TODAY`
4. If `new_status = delivered`: set `delivered_at = NOW()`
5. If `new_status = ready_for_pickup`: verify order is pickup type
6. If `new_status = picked_up`: set `delivered_at = NOW()` (income counted)
7. Update order status
8. If comment provided, update `staff_comment`
9. Create notifications for relevant users
10. Log info

### `api/orders/cancel_order.php` (POST)

**Allowed roles:** customer (own pending orders only), rider (own assigned/on_delivery — reverts to confirmed), staff, admin (any status)

**Input:** `order_id`, `reason` (required), `csrf_token`

**Logic:**
1. Fetch order
2. Validate permissions:
   - Customer: can only cancel their own orders at `pending` status
   - Rider: can cancel their assigned/on_delivery orders → order reverts to `confirmed`, rider_id set NULL
   - Staff/Admin: can cancel any order at any status
3. **Stock restoration:** If order status WAS confirmed/assigned/on_delivery (stock had been decremented), restore stock for each order_item:
   - For each order_item: UPDATE inventory SET stock_count = stock_count + quantity
   - If item was `out_of_stock` and now stock > 0, set status to `active`
4. Update order: status='cancelled', cancellation_reason, cancelled_by=session.user_id
5. If rider cancelled → set status='confirmed', rider_id=NULL, do NOT set cancelled status (it's a reassignment)
6. If customer cancelled → increment customer's cancellation_count. Check if count >= max_cancellation → if yes, set customer status='flagged'
7. Create notifications
8. Log info

### `api/orders/assign_rider.php` (POST)

**Allowed roles:** staff, admin

**Input:** `order_id`, `rider_id` (specific rider OR 'auto' for auto-assign), `csrf_token`

**Logic:**
1. Fetch order — must be status 'confirmed' (or 'assigned' for reassignment)
2. If `rider_id = 'auto'` → run Auto-Assign algorithm (see Key Algorithms section)
3. If specific rider_id → validate rider exists, is active, is available
4. If rider found:
   - Update order: rider_id, status='assigned'
   - Create `delivery_priority` entry for rider+order
   - If reassigning → notify old rider "Order reassigned", notify new rider "New delivery assigned"
   - Else → notify rider "New delivery assigned", notify customer "Rider assigned"
5. If no rider found (auto-assign failed) → return `{success: false, message: "No available riders. Order stays as Confirmed."}`
6. Log info

### `api/orders/bulk_update.php` (POST)

**Allowed roles:** staff, admin (and partially customer for bulk cancel of pending)

**Input:** `order_ids` (array), `action` ('confirm', 'cancel', 'assign_auto', 'on_delivery', 'delivered'), `reason` (for cancel), `csrf_token`

**Logic:**
1. Loop through order_ids
2. For each order, apply the action using the same logic as individual endpoints
3. For 'confirm': decrement stock for each, change status
4. For 'cancel': restore stock if needed, require reason
5. For 'assign_auto': run auto-assign for each confirmed order
6. Collect results: `{success: X, failed: Y, details: [...]}`
7. Create notifications for each
8. Log info

## IMPORTANT IMPLEMENTATION NOTES

1. **Every PHP file that is a protected page** must start by including `includes/auth_check.php` and then calling `require_role([...allowed roles...])`.

2. **Every API endpoint** must validate CSRF token on POST requests, check authentication, check role permissions, sanitize inputs, and return JSON.

3. **Every function** in every file must call `logger_function_entry()` at the start and `logger_function_exit()` before returning. Every significant operation must be logged with the appropriate level.

4. **Database queries** must use prepared statements (PDO) to prevent SQL injection. Use the `db_query()`, `db_fetch()`, `db_fetch_all()`, `db_insert()`, `db_update()`, `db_delete()` helper functions which automatically log queries.

5. **CSS variables** from admin color settings must be applied by PHP inline styles on the `<html>` element, read from the `settings` table. This way admin color changes take effect immediately.

6. **Dark mode** is determined by: (1) `force_dark_mode` setting — if ON, everyone is dark mode regardless; (2) If OFF, user's `user_preferences.dark_mode` value is used. The `<html>` element gets `data-theme="dark"` or `data-theme="light"` set by PHP.

7. **CSRF tokens** are generated per session and stored in `$_SESSION[CSRF_TOKEN_NAME]`. Every form and AJAX POST must include this token. Every API POST must validate it.

8. **File uploads** (item icons) must validate file type (images only), limit size (2MB), generate unique filenames, and store in `assets/uploads/items/`.

9. **The `receipt.php` page is PUBLIC** — no authentication required. It's accessed via the `receipt_token` in the URL. This is by design so customers can share receipts.

10. **When an account is soft-deleted** (status='deleted'), their orders remain in history. In order tables and history, show the username/name with a `[Deleted]` label. The account cannot log in.
